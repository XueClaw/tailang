# Tailang `.tai v0.3` 正式方案

## 状态

- 基于 PRD `v2.2-local`
- 本文档是 `.tai v0.3` 的正式语法方案
- 目标不是“中文化的西式语法”，而是“接近易语言自然书写习惯的文本源码语法”

## 核心定位

`.tai` 是 Tailang 的正式源码语言，不再等同于 JSON IR。

从现在起，仓库中存在三个不同层次：

1. `.meng`
   用户编写的自然语言工程源文件。
2. `.tai`
   由预编译器产出、由编译器消费、可审查、可编译、可版本控制的正式太语言源码。
3. 编译器内部 IR
   Rust 编译器解析 `.tai` 后使用的内部结构，不属于 `.tai` 语言本身。

旧版 JSON schema 不再代表 `.tai` 语言定义。
如果继续保留，只能视为以下之一：

- 兼容快照格式
- 传输格式
- 缓存格式
- 调试导出格式

## v0.3 设计目标

`.tai v0.3` 的目标是让语法更像真实易语言，而不是继续沿用“统一块壳 + 少量中文关键字”的过渡设计。

因此本版方案明确采用以下方向：

- 顶层和声明语句使用易语言风格点号关键字
- 过程控制使用语义化结束词，而不是统一块结束符
- 声明顺序尽量贴近自然中文
- 子程序、变量、参数、流程结构优先采用易语言式书写习惯

## 设计原则

`.tai v0.3` 必须满足以下原则：

1. 必须是中文语法
2. 结构声明必须采用点号前缀关键字
3. 不把 `{}` 当正式 `.tai` 语法
4. 不把统一 `.开始 / .结束` 当最终方向
5. 每类流程结构使用自己的语义化结束词

## 总体语法风格

### 1. 声明类语句带点号

例如：

- `.版本`
- `.支持库`
- `.程序集`
- `.程序集变量`
- `.子程序`
- `.参数`
- `.局部变量`
- `.常量`

### 2. 流程类语句使用自然中文

例如：

- `.如果`
- `.否则`
- `.如果结束`
- `.判断开始`
- `.判断`
- `.默认`
- `.判断结束`
- `.循环判断首`
- `.循环判断尾`
- `.计次循环首`
- `.变量循环首`
- `.变量循环尾`
- `.返回`
- `.跳出循环`
- `.到循环尾`

### 3. 表达式层尽量保留中文可读性

例如：

- `真 / 假 / 空`
- `并且 / 或者 / 非`
- `等于 / 不等于 / 大于 / 小于 / 大于或等于 / 小于或等于`

是否全面采用 `相加 / 相减 / 相乘 / 相除`，在 v0.3 中暂不强制；本版先允许符号运算与中文比较词并存，后续再统一表达式风格。

## 推荐声明语法

### 程序集

```tai
.程序集 登录模块
```

### 程序集变量

```tai
.程序集变量 当前用户, 文本型
```

### 子程序

推荐采用：

```tai
.子程序 登录, 文本型
.参数 邮箱, 文本型
.参数 密码, 文本型
```

解释：

- `.子程序 <名称>, <返回类型>`
- 参数不强行塞进括号，而是以独立声明行出现
- 这样更接近易语言的自然文本风格

### 局部变量

```tai
.局部变量 结果, 文本型
.局部变量 重试次数, 整数型
```

### 常量

```tai
.常量 默认重试次数 = 3
```

## 推荐流程语法

### 如果

```tai
.如果 邮箱 等于 ""
    .返回 "邮箱不能为空"
.如果结束
```

### 如果-否则

```tai
.如果 是否成功
    .返回 真
.否则
    .返回 假
.如果结束
```

### 判断

```tai
.判断开始 状态
.判断 "成功"
    返回 "ok"
.判断 "失败"
    返回 "fail"
.默认
    返回 "unknown"
.判断结束
```

### 循环

```tai
.循环判断首 计数 小于 10
    计数 = 计数 + 1
.循环判断尾
```

## `.代码` 与 `.待定`

### 代码补充

`.代码` 是补充实现块，不是 `.tai` 主执行语法本体。
它的职责是承载目标语言补充实现，而不是承担整个程序主体。

推荐形式：

```tai
.代码 Rust
    println!("hello");
.代码结束
```

说明：

- v0.3 推荐 `.代码 ...` / `.代码结束`
- 不再推荐 `.代码 .开始 ... .结束`
- 代码补充块是特例，应拥有自己的结束词

### 待定项

```tai
.待定 规则, "缺少密码复杂度规则"
```

## 与旧草案的关键差异

`.tai v0.3` 明确放弃以下过渡思路：

- 统一 `.开始 / .结束` 作为所有结构块的正式写法
- 把 `.子程序` 写成 `函数名(参数)` 这种偏西式签名
- 让 `.实现` 成为“唯一可执行块容器”

`.tai v0.3` 转向以下正式方向：

- 顶层声明直接书写，不再强依赖统一块壳
- 子程序参数与变量以独立声明行表达
- 流程结构使用 `.如果结束 / .判断结束 / .循环判断尾` 等语义化结束词
- 代码补充块使用自己的结束词，如 `.代码结束`

## 语义边界

### `.tai` 应该承担的

- 程序结构
- 子程序定义
- 参数与变量声明
- 控制流
- 返回与表达式
- 代码补充块引用
- 待定项标记

### `.tai` 不应该继续承担的

- 被当作 JSON 快照
- 被当作纯说明文件
- 把正式执行语法全部外包给目标语言代码块

## 编译流水线

目标流水线如下：

`.meng` -> Provider 预编译 -> `.tai` 文本源码 -> Tailang parser -> Tailang 语义 IR -> 原生 / Web / API 目标

## v0.3 实施优先级

建议按以下顺序落地实现：

1. 顶层声明语法
   - `.版本`
   - `.目标平台`
   - `.程序集`
   - `.子程序`
   - `.参数`
   - `.局部变量`

   - `.如果`
   - `.否则`
   - `.如果结束`
3. 返回与赋值
   - `.返回`
   - 赋值表达式
4. 循环语法
   - `.循环判断首`
   - `.循环判断尾`
5. 判断语法
   - `.判断开始`
   - `.判断`
   - `.默认`
   - `.判断结束`
6. 代码补充块
   - `.代码`
   - `.代码结束`

## 当前仓库需要消除的旧假设

- `.tai == JSON IR`
- `.tai schema == 语言规范`
- `.tai` 只是“结构说明文件”
- `.tai` 必须依赖统一块界定符才可解析

## 当前仓库必须建立的新假设

- `.tai` 是正式文本源码语言
- `.tai` 的正式风格应接近真实易语言书写习惯
- `compiler/` 负责真正的 `.tai` 解析与编译
- 编译器 IR 与 `.tai` 语言层清晰分离
